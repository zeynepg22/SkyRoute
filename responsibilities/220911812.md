## Database Schema Contributions

I designed and implemented the main database schema for the platform.

### Tables Implemented

| Table | Purpose |
|---|---|
| `roles` | Stores user roles such as student, instructor, and admin |
| `users` | Stores platform users and authentication-related user information |
| `courses` | Stores course information created by instructors |
| `lessons` | Stores lesson content and embedded video URLs |
| `enrollments` | Stores which student enrolled in which course |
| `lesson_progress` | Tracks completed lessons for each student |
| `social_accounts` | Supports social login providers |
| `two_factor_methods` | Stores 2FA method configuration |
| `backup_codes` | Stores hashed backup recovery codes |
| `audit_logs` | Stores important user/system actions for traceability |

## Relationship Design

I implemented relational database connections using foreign keys and SQLModel relationships.

### Main Relationships

| Relationship | Description |
|---|---|
| `roles → users` | One role can belong to many users |
| `users → courses` | One instructor can create many courses |
| `courses → lessons` | One course can contain many lessons |
| `users → enrollments` | One student can enroll in many courses |
| `courses → enrollments` | One course can have many enrolled students |
| `users → lesson_progress` | One student can have many lesson progress records |
| `lessons → lesson_progress` | One lesson can have many progress records |
| `users → social_accounts` | One user can connect multiple social login providers |
| `users → two_factor_methods` | One user can enable multiple 2FA methods |
| `users → backup_codes` | One user can have multiple recovery codes |
| `users → audit_logs` | User actions can be tracked in audit logs |

## Data Integrity Rules

I added database constraints to protect data consistency.

| Constraint | Purpose |
|---|---|
| Unique user email | Prevents duplicate user accounts with the same email |
| Unique role name | Prevents duplicate role definitions |
| Unique enrollment per user/course | Prevents a student from enrolling in the same course more than once |
| Unique progress per user/lesson | Prevents duplicate lesson completion records |
| Unique lesson order per course | Prevents two lessons in the same course from having the same order index |
| Foreign keys | Ensures relationships between tables stay valid |

## Progress Calculation Logic

I implemented the course progress calculation logic.

### Formula

```text
Progress = completed lessons / total lessons × 100
```

### Example

If a course has 3 lessons and the student completed 2 lessons:
```text
2 / 3 × 100 = 66.67%
```

### Implemented Logic

The logic is implemented in:
```text
backend/services/progress_service.py
```

The function calculates a student's course completion percentage by comparing completed lesson records with the total number of lessons in the selected course.
