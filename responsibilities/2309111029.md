## Authentication Implementation

### Basic Authentication (Email/Password)

| Feature | Implementation |
|---|---|
| Registration | `POST /auth/register` — validates input, hashes password with bcrypt, creates user with default student role |
| Login | `POST /auth/login` — verifies email/password, checks 2FA status, issues JWT or temp token |
| Password Hashing | bcrypt via passlib CryptContext |
| JWT Access Token | 1-hour expiry, contains user_id + role in payload, signed with HS256 |
| JWT Refresh Token | 7-day expiry, used to get new access tokens without re-login |
| Token Refresh | `POST /auth/refresh` — validates refresh token, issues new token pair |
| Current User | `GET /auth/me` — returns authenticated user profile with role, 2FA status, social providers |

### JWT Token Structure

```
Header:  {"alg": "HS256", "typ": "JWT"}
Payload: {"sub": "user_id", "role": "student", "type": "access", "exp": ..., "iat": ...}
Signature: HMAC-SHA256(header.payload, SECRET_KEY)
```

### Protected Routes

Implemented via FastAPI dependency injection:
- `get_current_user` — extracts and validates JWT from Authorization header
- `require_role("admin")` — restricts endpoint to specific roles
- Checks user is_active status on every authenticated request

## Two-Factor Authentication (4 Methods)

### Method 1 — TOTP (Google Authenticator)

| Feature | Detail |
|---|---|
| Setup | `POST /auth/2fa/setup/totp` — generates secret, returns provisioning URI for QR code |
| Verification | 6-digit time-based code verified with pyotp, 1-window tolerance |
| Storage | Secret stored in `two_factor_methods` table |

### Method 2 — Email OTP

| Feature | Detail |
|---|---|
| Setup | `POST /auth/2fa/setup/email-otp` — enables email OTP for user |
| Send OTP | `POST /auth/send-email-otp` — generates 6-digit code, 5-minute expiry |
| Verification | Code verified against in-memory store |
| Development | OTP printed to console when SMTP is not configured |

### Method 3 — Backup Recovery Codes

| Feature | Detail |
|---|---|
| Generation | `POST /auth/2fa/backup-codes` — generates 8 codes in XXXX-XXXX format |
| Storage | Codes are bcrypt-hashed in `backup_codes` table |
| Usage | Single-use — code marked as used after successful verification |

### Method 4 — Trusted Device / Remember Device

| Feature | Detail |
|---|---|
| Setup | `POST /auth/2fa/setup/trusted-device` — returns a device token |
| Storage | Token hash stored in `two_factor_methods` table |
| Usage | Client sends device token during 2FA step to skip code entry |

### 2FA Login Flow

```
1. POST /auth/login (email + password)
2. If 2FA enabled → returns {requires_2fa: true, temp_token, available_methods}
3. POST /auth/verify-2fa (temp_token + method + code)
4. If verified → returns {access_token, refresh_token}
```

## Social Login (4 Providers)

All providers use OAuth 2.0 Authorization Code Flow.

| Provider | Auth URL | Scopes |
|---|---|---|
| Google | accounts.google.com/o/oauth2/v2/auth | openid email profile |
| GitHub | github.com/login/oauth/authorize | user:email |
| Discord | discord.com/api/oauth2/authorize | identify email |
| Facebook | facebook.com/v18.0/dialog/oauth | email public_profile |

### OAuth Flow

```
1. GET /auth/social/{provider} → redirects user to provider login page
2. User authenticates and grants permission
3. Provider redirects to GET /auth/callback/{provider}?code=...
4. Backend exchanges code for access token (server-to-server)
5. Backend fetches user profile from provider API
6. Backend creates/links user in database
7. Backend redirects to frontend with JWT tokens
```

## Authorization (Role-Based Access Control)

### Roles

| Role | Permissions |
|---|---|
| Student | Enroll in courses, view lessons, complete lessons, track progress |
| Instructor | Create/edit own courses and lessons, plus all student permissions |
| Admin | Manage all users, change roles, activate/deactivate users, manage all courses |

---

## Admin Endpoints

| Endpoint | Purpose |
|---|---|
| `GET /admin/users` | List all users with roles |
| `GET /admin/users/{id}` | Get single user details |
| `PUT /admin/users/{id}/role` | Change user role |
| `PUT /admin/users/{id}/status` | Activate or deactivate user account |
| `GET /admin/audit-logs` | View recent audit log entries |

### Safety Checks

- Admin cannot change their own role
- Admin cannot deactivate their own account
- All admin actions are logged in `audit_logs` table

## Security Implementation

### Rate Limiting (slowapi)

- Login endpoint: 5 requests/minute per user (JWT sub) or IP
- Returns 429 Too Many Requests with Retry-After header

### Security Headers

| Header | Value | Purpose |
|---|---|---|
| X-Frame-Options | DENY | Prevents clickjacking |
| X-Content-Type-Options | nosniff | Prevents MIME-type sniffing |
| X-XSS-Protection | 1; mode=block | Legacy XSS filter |
| Strict-Transport-Security | max-age=31536000 | Forces HTTPS |
| Content-Security-Policy | default-src 'self' | Prevents script injection |
| Referrer-Policy | strict-origin-when-cross-origin | Controls referrer info |

### CORS Configuration

- Specific origins only (no wildcard `*`)
- Frontend URL loaded from environment variable

### Environment Variables

- All secrets stored in `.env` file
- `.env` added to `.gitignore`
